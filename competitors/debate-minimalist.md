# 집중론자 관점: VulChk 제품 전략 토론

> 작성일: 2026-03-02
> 관점: "작지만 날카로운 도구"로서의 VulChk

---

## 핵심 주장

**VulChk의 정체성은 "바이브 코더를 위한 경량 보안 플러그인"이다.**

`npx vulchk`로 설치하고, Claude Code 안에서 슬래시 명령어로 즉시 실행한다.
계정 생성 없음, 플랫폼 연동 없음, Docker 설정 없음.
이 즉시성과 경량성이 VulChk의 존재 이유이며, 이것을 해치는 기능 확장은 자해 행위다.

바이브 코더는 보안 전문가가 아니다. 그들이 원하는 것은 "이 코드 안전해?"라는 질문에 대한 빠르고 명확한 답이지, STRIDE 매트릭스나 CI/CD 파이프라인 설정이 아니다.

---

## 도입 불필요 (경쟁사가 하지만 VulChk가 하면 안 되는 것)

### 1. STRIDE 위협 모델링

**SecureVibes의 4단계 STRIDE 파이프라인은 바이브 코더에게 과잉이다.**

- SecureVibes는 Assessment → Threat Modeling → Code Review → Report의 순차 파이프라인을 돌린다. 스캔당 $2-8, 시간도 길다.
- STRIDE의 6개 카테고리(Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege)를 바이브 코더가 읽고 이해하고 조치할 수 있는가? 그들은 `THREAT_MODEL.json`을 열어보지 않을 것이다.
- VulChk의 접근법이 더 낫다: **"이 줄에 SQL Injection이 있다. 이렇게 고쳐라."** 실행 가능한 한 줄 피드백이 30페이지 위협 모델보다 바이브 코더에게 100배 유용하다.
- STRIDE를 도입하면 분석 시간이 2-3배 늘어나고, 순차 실행이 강제되어 현재의 5개 에이전트 병렬 실행 장점이 사라진다.

**결론:** STRIDE 위협 모델링은 보안 컨설턴트를 위한 도구다. 바이브 코더를 위한 도구가 아니다.

### 2. CI/CD 네이티브 통합 (GitHub Actions / GitLab CI)

**"Claude Code 안에서 즉시 실행"이 VulChk의 핵심 UX인데, 왜 CI/CD로 나가는가?**

- Aikido의 CI Gating, SecureVibes의 PR Review, Claude Code Security의 GitHub Action — 모두 CI/CD 통합을 제공한다. 그런데 이것들은 설정이 필요하다:
  - GitHub Actions YAML 작성
  - API 키를 Secrets에 등록
  - 실패 정책(fail/warn) 결정
  - Docker 이미지 빌드 또는 Runner 환경 설정
- 바이브 코더는 GitHub Actions YAML을 쓰지 않는다. Replit이나 Bolt에서 코드를 만들어 바로 배포한다. CI/CD 파이프라인 자체가 그들의 워크플로우에 존재하지 않는다.
- VulChk가 CI/CD에 들어가는 순간, "제로 설정"이라는 핵심 가치가 깨진다. "설정 안내 문서 10페이지"가 필요한 보안 도구는 바이브 코더가 쓰지 않는다.

**결론:** CI/CD 통합은 바이브 코더의 워크플로우 밖에 있다. VulChk의 전장이 아니다.

### 3. 커밋 레벨 지속 감시

**Codex Security가 커밋마다 자동 스캔한다. 멋지다. 그리고 토큰 비용이 폭발한다.**

- 커밋 레벨 감시는 "모든 커밋 × 전체 코드베이스 컨텍스트"를 LLM에 보내야 한다. 하루 20-30 커밋을 하는 바이브 코더의 경우, 하루 API 비용만 $50-100이 될 수 있다.
- Codex Security는 GPT-5 기반에 Private Beta이고 가격을 공개하지 않았다. 그들은 엔터프라이즈 보안팀을 타겟으로 한다. VulChk의 타겟과 완전히 다르다.
- 바이브 코더에게 적합한 모델은 **온디맨드 스캔**이다. "배포 전에 한 번 돌려보자"가 그들의 멘탈 모델이다. 항상 켜진 감시자가 아니라 필요할 때 부르는 전문가.
- VulChk의 현재 증분 업데이트(커밋 해시 기반 diff)가 이미 이 문제를 올바르게 해결하고 있다.

**결론:** 온디맨드 스캔이 바이브 코더의 비용 구조와 워크플로우에 맞다. 커밋 감시는 과잉 설계다.

### 4. 올인원 플랫폼 지향 (Aikido 따라가기)

**Aikido는 SAST + SCA + DAST + Container + IaC + CSPM + Runtime + AI Pentest를 한다. 그리고 $350/월부터 시작한다.**

- Aikido의 9가지 보안 스캔을 따라가려면 VulChk는 더 이상 "경량 CLI 플러그인"이 아니다. 별도의 클라우드 인프라, 대시보드, 계정 시스템이 필요하다.
- CSPM(Cloud Security Posture Management)? 바이브 코더가 AWS IAM 정책을 관리하지 않는다. Runtime 보호(RASP)? 바이브 코더가 미들웨어를 커스텀 설정하지 않는다.
- Aikido가 성공한 이유는 "통합"이 아니라 "각 기능의 품질 + AI AutoTriage"이다. 기능 수가 아니라 기능 깊이가 중요하다.
- VulChk가 모든 것을 하려 하면, 아무것도 잘 하지 못한다. 현재 VulChk의 강점(SCA + Secrets + DAST + Code Pattern)에 집중하는 것이 옳다.

**결론:** 올인원 플랫폼은 VulChk의 정체성을 파괴한다. "작지만 날카로운 도구"가 우리의 포지션이다.

### 5. 다중 언어 대규모 확대 (30개+ 언어 지원)

**Semgrep은 30개 언어를 지원한다. 하지만 Semgrep은 AST 패턴 매칭 엔진이다. VulChk는 LLM 기반이다.**

- VulChk의 LLM(Claude)은 이미 대부분의 프로그래밍 언어를 "이해"한다. 언어 지원은 추가 개발 비용이 아니라 LLM의 내재적 능력이다.
- 진짜 문제는 다르다: **바이브 코더가 실제로 쓰는 언어는 JavaScript/TypeScript, Python, 그리고 약간의 Go 정도다.** Replit에서 Rust로 코딩하는 바이브 코더는 극소수다.
- 30개 언어 지원을 광고하는 것은 마케팅 차원에서는 의미 있지만, 각 언어의 에코시스템(패키지 매니저, 의존성 구조, 일반적 취약점 패턴)에 대한 에이전트 커스터마이징은 엄청난 유지보수 부담이다.
- dependency-auditor가 OSV API를 쓰므로 기본적인 SCA는 이미 언어 무관하게 동작한다. 추가 투자가 필요한 부분이 명확하지 않다.

**결론:** JS/TS + Python을 완벽하게 지원하는 것이 30개 언어를 어중간하게 지원하는 것보다 낫다.

### 6. 맬웨어 분석

**Codex Security가 2026년 2월에 맬웨어 분석을 추가했다. VulChk가 할 이유는 전혀 없다.**

- 맬웨어 분석은 보안 연구원과 SOC 팀의 영역이다. 바이브 코더가 맬웨어 샘플을 분석할 일이 없다.
- VulChk의 컨텍스트는 "내가 작성한/생성한 코드가 안전한가?"이다. "이 바이너리가 악성인가?"가 아니다.
- 완전히 다른 타겟 사용자, 완전히 다른 기술 스택이 필요한 기능이다.

**결론:** 논의할 가치조차 없다. VulChk의 범위 밖이다.

## 조건부 도입 (가치가 있지만 신중해야 하는 것)

### 1. 자동 수정(AutoFix) 제안

**Aikido의 AutoFix, Snyk의 DeepCode AI Fix, Claude Code Security의 패치 제안 — 매력적이지만 위험하다.**

- Snyk 자체가 경고한다: "AI가 생성한 수정 코드 자체가 취약할 수 있다." AI 생성 코드는 XSS 취약점 도입 확률이 2.74배 높다.
- 바이브 코더는 패치를 제대로 검증할 능력이 없다. 그들이 AI 코드를 신뢰하고 그대로 적용하면, AutoFix가 새로운 취약점 벡터가 된다.
- **조건부 도입 방안:** "수정 코드"가 아니라 "수정 방향"을 제안하는 것은 가능하다. "이 줄에서 `parameterized query`를 사용하세요" 수준의 가이던스는 현재 리포트에서도 이미 부분적으로 하고 있다. 코드 패치 자동 적용은 하지 않는다.

**도입 조건:**
- 코드 패치 자동 적용 금지 (수정 "방향" 제안만)
- 현재 리포트의 "수정 가이드" 품질 향상에 집중
- AI 패치가 새 취약점을 도입하지 않는다는 검증 없이 AutoFix PR 기능을 추가하지 않음

### 2. 샌드박스 검증

**Codex Security는 격리 환경에서 취약점을 실제로 트리거하여 false positive를 줄인다. 원리는 좋다.**

- 문제는 인프라다. 샌드박스 환경을 만들려면 Docker, 가상 환경, 네트워크 격리 등 복잡한 설정이 필요하다.
- VulChk의 현재 HackSimulator는 `curl` 기반 프로브로 실제 HTTP 요청을 보낸다. 이것은 "가벼운 샌드박스 검증"이다. 완전한 격리 환경은 아니지만, 바이브 코더의 일반적인 웹앱에 대해서는 충분하다.
- **조건부 도입 방안:** 전용 샌드박스 인프라를 구축하는 대신, 기존 HackSimulator의 검증 깊이를 강화한다. 예: SQLi 탐지 시 실제 에러 응답을 파싱하여 확인, XSS 탐지 시 반사 여부 확인 등.

**도입 조건:**
- 추가 인프라(Docker, VM) 요구 없음
- 기존 curl/HTTP 기반 프로브의 검증 정확도 향상에 집중
- Playwright 브라우저 테스트를 활용한 DOM 기반 검증 확대

### 3. JSON 구조화 출력

**SecureVibes는 `VULNERABILITIES.json`, `THREAT_MODEL.json` 등 구조화된 JSON을 출력한다. VulChk는 마크다운만 출력한다.**

- 마크다운은 사람이 읽기 좋다. 바이브 코더는 리포트를 "읽는다." JSON을 프로그래밍적으로 처리하는 것은 보안 자동화 파이프라인의 영역이다.
- 그러나 향후 CI/CD 연동이나 다른 도구와의 통합을 고려하면, JSON 출력은 부수적 비용이 낮은 기능이다.
- **조건부 도입 방안:** 마크다운 리포트를 기본으로 유지하되, `--format json` 플래그로 JSON 출력을 선택적으로 제공한다. 핵심 UX(마크다운 리포트)를 해치지 않으면서 확장성을 확보한다.

**도입 조건:**
- 기본 출력은 마크다운 유지
- JSON은 옵션 플래그로만 제공
- JSON 스키마 유지보수 부담이 과도해지면 제거

### 4. CWE 매핑

**SecureVibes는 모든 취약점에 CWE ID를 태깅한다. Snyk도, Semgrep도 한다.**

- CWE 매핑은 업계 표준이며, 보안 리포트의 신뢰도를 높인다.
- 바이브 코더가 CWE-89를 검색하지는 않겠지만, "이건 CWE-89 (SQL Injection)에 해당합니다"라는 한 줄이 리포트의 전문성을 올려준다.
- LLM이 이미 CWE를 알고 있으므로 구현 비용이 낮다. 에이전트 프롬프트에 "각 취약점에 CWE ID를 포함하라"는 지시만 추가하면 된다.

**도입 조건:**
- 에이전트 프롬프트 수정만으로 구현 (추가 API/DB 불필요)
- 리포트 가독성을 해치지 않는 선에서 표시 (괄호 안에 CWE-XXX 형태)

## 도입 필요 (경쟁사에서 배워야 할 것)

### 1. False Positive 자동 필터링

**이것은 VulChk의 가장 큰 약점이다. 반드시 개선해야 한다.**

- Aikido의 AI AutoTriage는 95% false positive 감소를 달성했다. Snyk Code는 0.08% false positive 비율을 주장한다. Semgrep Assistant는 SAST 결과의 20%를 false positive로 필터링하며 95% 사용자 동의율을 보인다.
- VulChk는 현재 LLM의 판단에 의존하지만, 체계적인 FP 필터링 메커니즘이 없다. code-pattern-scanner의 4단계 CoT 추론이 있지만, 이것은 탐지 정확도이지 사후 필터링이 아니다.
- false positive가 많으면 바이브 코더는 리포트를 신뢰하지 않게 되고, 도구를 안 쓰게 된다. **신뢰가 모든 것이다.**
- Claude Code Security의 "적대적 자기검증(Adversarial Verification Pass)"은 VulChk에서도 구현 가능하다. 각 에이전트가 결과를 반환한 후, SKILL.md의 병합 단계에서 LLM이 "이 발견이 진짜인가?"를 한 번 더 검증하는 것이다.

**구현 방안:**
- codeinspector SKILL.md의 병합 단계에서 적대적 검증 패스 추가
- "이 취약점이 실제로 악용 가능한가? 도달 가능한 코드 경로인가?"를 LLM에게 재확인
- 신뢰도(confidence) 등급을 각 발견 사항에 추가
- 추가 에이전트나 인프라 불필요 — 기존 SKILL.md 프롬프트 수정만으로 가능

### 2. 에이전틱 앱 보안 탐지

**SecureVibes는 LLM API 호출/에이전트 프레임워크 사용을 자동 감지하고, OWASP ASI 위협을 자동 포함시킨다. 이것은 VulChk의 타겟 사용자에게 정확히 필요한 기능이다.**

- 바이브 코더의 상당수가 AI 앱을 만든다. Claude API, OpenAI API, LangChain, CrewAI 등을 사용하는 프로젝트가 점점 많아진다.
- 프롬프트 인젝션, 모델 다운그레이드, 도구 매개변수 검증 미흡 — 이것들은 전통적 OWASP Top 10에 포함되지 않지만, 에이전틱 앱에서 가장 흔한 취약점이다.
- VulChk의 code-pattern-scanner가 이미 OWASP Top 10을 탐지하지만, AI/에이전트 특화 위협은 명시적으로 다루지 않는다.
- 이 기능은 VulChk의 핵심 가치(바이브 코딩 특화)를 직접적으로 강화한다. 바이브 코더가 가장 많이 만드는 것이 AI 앱이기 때문이다.

**구현 방안:**
- code-pattern-scanner 프롬프트에 OWASP ASI(Agentic Security Issues) 탐지 규칙 추가
- 프로젝트에서 LLM API 호출 패턴 감지 시 자동으로 에이전틱 보안 규칙 활성화
- 추가 에이전트 불필요 — 기존 code-pattern-scanner 확장으로 충분

### 3. PR/Diff 기반 증분 리뷰 강화

**SecureVibes v0.4.0의 PR Review 워크플로우에서 배울 점이 있다. 단, CI/CD가 아닌 Claude Code 내부에서 실행하는 방식으로.**

- VulChk의 현재 증분 업데이트는 커밋 해시 기반 diff로 동작한다. 이것은 좋다.
- 개선할 수 있는 점: SecureVibes처럼 "저위험 diff(문서, 테스트, 설정만 변경)에 대한 자동 트리아지"를 추가하면 비용을 절감할 수 있다.
- 또한 "이전 스캔 대비 신규/해결/변경된 취약점"을 명확하게 표시하면 바이브 코더가 변화를 빠르게 파악할 수 있다.
- 이것은 CI/CD 통합이 아니다. Claude Code 안에서 `/vulchk.codeinspector` 실행 시 자동으로 diff를 분석하는 것이다. VulChk의 현재 아키텍처와 완벽하게 호환된다.

**구현 방안:**
- codeinspector SKILL.md에서 diff 분석 시 저위험 파일 자동 스킵 로직 추가
- 리포트에 "신규/해결/변경" 섹션 추가
- 기존 증분 업데이트 메커니즘 위에 구축

### 4. 비용 가시성

**SecureVibes는 모델별 비용 비교($0.15 ~ $7.64)를 투명하게 공개한다. VulChk는 비용에 대한 정보가 없다.**

- 바이브 코더에게 비용은 중요한 관심사다. 특히 Claude Code의 토큰 기반 과금 모델에서.
- "이 스캔에 대략 얼마가 들었다"는 정보는 사용자 신뢰를 높이고, 불필요한 재실행을 방지한다.
- Claude Code의 토큰 사용량을 정확히 추적하기는 어렵지만, 에이전트별 모델과 대략적인 토큰 사용량을 리포트 말미에 표시하는 것은 가능하다.

**구현 방안:**
- 리포트 말미에 "분석 요약" 섹션 추가 (사용된 에이전트 수, 각 에이전트의 모델, 대략적 실행 정보)
- 정확한 비용 추적보다는 투명성 확보에 초점

## 마무리: 집중론자의 원칙

1. **모든 기능 추가 전에 물어라**: "이것이 바이브 코더가 Claude Code 안에서 즉시 쓸 수 있는가?"
2. **추가 인프라가 필요하면 하지 마라**: Docker, 클라우드 계정, CI 설정이 필요한 기능은 VulChk의 DNA에 맞지 않는다.
3. **기능 수가 아니라 기능 깊이로 승부하라**: 5개 에이전트가 각자의 영역에서 최고 수준의 정확도를 달성하는 것이 15개 에이전트가 어중간한 품질을 제공하는 것보다 낫다.
4. **경쟁사를 따라가지 마라, 사용자를 따라가라**: Aikido가 9개 스캐너를 가졌다고 VulChk도 9개가 필요한 것이 아니다. 바이브 코더가 실제로 필요로 하는 것에 집중하라.

VulChk의 경쟁 우위는 **"가장 많은 기능"이 아니라 "가장 낮은 마찰"**이다. 이것을 지켜야 한다.
